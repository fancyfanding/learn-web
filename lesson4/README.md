# 项目亮点（整体的讲述过程控制在6分钟以内）

## 心天地

1. 项目亮点1-中后台数据密集型表格的性能治理：

   S：需要运营同学高频查看一些信息，如问卷完成率，需要支持筛选、排序，当数据量上来之后，页面会出现卡顿、滚动不流畅，筛选延迟等问题

   T：针对以上问题，为了保证数据量较大时，能够交互流畅，筛选能够更加迅速，页面不卡顿

   A：

   1. 定位瓶颈，利用chrome performance 发现是因为dom节点过多，筛选条件触发了全量的计算和渲染；
   2. 减少渲染的开销就是核心的卡点，对长列表进行按需加载，虚拟滚动，把dom的数量控制在一定的区间，做服务端的分页，减少单页渲染的规模，减少重复计算（**防抖，节流**）,在筛选的时候加入debounce，防止每次输入的时候触发请求
   3. 交互稳定性，加入loding的状态，错误的兜底(try catch)

   R：提升了系统稳定性、流畅性，当数据量上来之后也能够流畅运行（量化指标）
2. 项目亮点2-小程序的模块设计：

   S：小程序作为一个内容社区和互动平台，平台包含的资源比较多，同质化的页面比较多，需要对小程序进行整体的组件设计

   T：将内容流详情互动，做成高内聚，可复用的模块

   A：页面状态模型的拆分：

   1. 包含文章、电台、测评、帖子等类型，每种类型下都设置有评论区，于是将评论区的消息进行了解耦，单列成一个组件，方便不同页面的复用，减少了重复代码
   2. 针对于每个模块的内容，采用卡片的形式进行展示，在个人中心中，又设置有浏览历史、收藏、点赞等模块，用户的阅读、使用记录将进行记录，于是将每个板块的内容模块进行了统一设计，分别给予不同状态，保持卡片列表的内容一致性但又不影响各页面的功能

   R：形成了从内容到内部转化的完整的链路，用户的操作响应会更快，并且后续新的内容加入的成本显著降低，提高了项目的可维护性
3. 项目亮点3-审核流和数据权限：

   S：社区内容需要一定的审核，用户投稿不会立即展示出来，需要由管理员进行PC审核，然后再进行公布，不符合要求的会进行驳回，并标注原因

   T：设计一套可以拓展的审核工作流，包含了状态的流转到通过/驳回，权限的边界

   A：

   1. 建立一个状态机，将整个内容的状态定义为特定的值，draft（草稿箱），pending（待审核），approve（成功），reject（失败）
   2. 权限控制：通过后端的数据角色情况来控制路由、页面的跳转、按钮的显示与否、数据的获取，防止越权
